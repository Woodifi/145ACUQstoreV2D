# QStore IMS — v2.1 Backlog

Tasks deferred from v2.0 development. Listed roughly in priority order for
v2.1 — items at the top should be done first when v2.1 work starts.

## Critical for unit deployment

### CO recovery flow
**Problem.** If the CO forgets their PIN and there's no second admin user,
recovery requires wiping IndexedDB — which is data loss. v2.0 has no
in-app recovery path.

**Direction.** Two options to consider:
- "Forgot PIN" recovery via a CO-only emergency code generated and stored
  at install time (printed/written down). Entering it resets the PIN.
- Mandatory second admin: refuse to let the unit operate with a single
  CO-role account. Force creation of a second account (deputy CO or QM
  with admin permissions) before initial setup completes.

Worth flagging that recovery via cloud sync is also possible if the unit
has cloud configured — they could pull a known-good snapshot from another
device. But this only helps units that have cloud sync.

### Manual data export and import buttons
**Problem.** v2.0 has working `Storage.exportAll()` and `Storage.importAll()`
under the hood — `loadFromCloud` already uses them — but no Settings UI for
"export to local file" or "import from local file". For standalone-no-cloud
deployments, this is the only backup path. Without it, a USB-stick-only
unit has no way to back up.

**Direction.** Two buttons in Settings → "Data" section. Export downloads
the snapshot as JSON. Import accepts a file via `<input type="file">`,
parses it, runs `Storage.importAll()` after the same OVERWRITE confirmation
flow used for `loadFromCloud`.

### Unit branding (name, code, crest)
**Problem.** The `unitName` and `unitCode` settings exist in the schema and
display in the header, but Settings has no UI to edit them. Default is
"QStore IMS" / blank. For productisation each unit needs to set its own.

**Direction.** Settings → "Unit details" section. Inputs for unit name,
unit code (e.g. "1 ACTU"). Optional: upload a unit crest as a Blob to
settings, render in the header next to the brand name.

## Cloud sync improvements

### Bump MSAL Browser to 5.x
**Problem.** v2.0 pins `@azure/msal-browser@2.38.3`. `npm install` prints a
deprecation warning. The library works but won't get security fixes from
Microsoft.

**Direction.** Migration guide is well documented. Core API surface
(`PublicClientApplication`, `loginRedirect`, `acquireTokenSilent`, etc.)
is stable. Most breaking changes are in TypeScript types and edge-case
response shapes. Treat as an isolated task with focused testing — don't
combine with other work.

### Conflict detection on cloud sync
**Problem.** v2.0 uses last-write-wins. If two QMs edit the same Q-Store
from different devices simultaneously, one will silently lose work. The
`cloudSync.pushedAt`/`pushedBy` annotation in the snapshot helps a careful
QM notice, but only if they look.

**Direction.** Before each push, fetch the current cloud `lastModified`
timestamp via Graph metadata. If it's newer than the timestamp recorded
when we last pulled, present a merge dialog ("cloud has changes from
[user] at [time] — keep local, keep cloud, or compare?"). Compare view
needs UX work; v2.2+ probably.

### Encrypt cloud blob client-side
**Problem.** Cloud blob is unencrypted JSON. Anyone with OneDrive access
sees every PIN hash, cadet name, service number. PIN hashes are argon2id
so they're hard to crack, but the rest is plain.

**Direction.** Encrypt the snapshot client-side with a unit master key
before upload. Master key derived from a passphrase set during cloud
configuration. Key never leaves the device. Trade-off: forgotten passphrase
means the cloud blob is unrecoverable. Worth doing once the simpler issues
are resolved — meaningful security gain for not much code.

### Selective sync — split the cloud blob
**Problem.** v2.0 pushes the entire snapshot (items + photos + audit + …)
on every change. For a 200-item Q-Store with 50 photos, that's a few MB
per push. Wasteful and slow.

**Direction.** Split into multiple OneDrive files: `data.json` (small,
syncs frequently), `audit.jsonl` (append-only, syncs delta), `photos/<id>`
(per-photo files, only sync changed ones). Bigger architectural change —
the CloudProvider interface needs to grow read/write per-key methods.
Worthwhile when units start hitting the limit.

## UI / UX

### Toast component to replace `alert()`
**Problem.** Several places (item-deleted-by-other-tab errors, sign-in
failures) use `alert()`. Functional but ugly and modal — blocks the UI
until dismissed.

**Direction.** Generalise the `flash` element used by the settings page
into a proper `toast.js` module. Stack multiple toasts. Auto-dismiss with
configurable timeout. Variants for info/success/warn/error.

### Mobile-specific layout for inventory table
**Problem.** Inventory table works on phones via horizontal scroll, but
ten columns is genuinely hard to read on a 6-inch screen. v1 had a
separate mobile card layout for inventory.

**Direction.** Below ~600px viewport, switch the inventory list to a card
layout: NSN + name + photo + condition badge as the visible header,
expandable for the rest. Same data, different presentation.

### Pagination for large item lists
**Problem.** The inventory page renders every row. With 1000+ items the
table becomes long. Realistic AAC unit Q-Stores are 50-200 items so this
isn't urgent, but it's a real ceiling.

**Direction.** Either virtual scrolling or simple pagination (50 per page).
Virtual scrolling is more work but better UX once you have it. Pagination
is enough for the data sizes we expect.

### "Unsaved changes" prompt on edit modal close
**Problem.** Closing the item edit modal with unsaved changes silently
discards them. Easy to lose work by misclick.

**Direction.** Track form dirty state. On close attempt with dirty form,
prompt: "You have unsaved changes. Discard them?" v1 also didn't have
this; not urgent but adds polish.

## Other features the user mentioned earlier

### Loans (issue/return) page
Second-largest UI surface after inventory. Issue an item to a cadet,
record return, partial returns, overdue tracking. Patterns are mechanical
replication of inventory page now that those are proven.

### Cadets / personnel page
Smaller than loans. CRUD for cadets — name, rank (with the canonical
AAC rank list we established), service number, parent contact, notes.

### Audit log viewer
Read-only page. Filter by date, action, user. Show the chain verification
status (green if `verify()` returns ok, red with broken-at-seq if not).
Export to CSV for reporting.

### Stocktake mode
Walk through the inventory item-by-item, scan or enter quantity counted,
record discrepancies. Generate a stocktake report.

### AB189 PDF generation
The form-fill flow from v1 — generate AB189 stocktake forms with item
data filled in. Needs jsPDF (~150KB bundled).

### QR code generation
Generate QR labels for items linking to NSN/name. Needs a QR library
(qrcode.js or similar, ~30KB).

## Process / infrastructure

### Tests for cloud.js and sync.js
**Problem.** v2.0 ships these without tests because real OAuth requires
user interaction. The `_setProvider` hook in cloud.js exists for this.

**Direction.** Mock CloudProvider that simulates read/write/sign-in
without actually calling Microsoft. Tests cover: debounce coalescing,
manual sync, load-from-cloud overwrite flow, error path on auth fail,
audit chain reconstruction after import.

### CI/build automation
**Problem.** Currently `npm run build` is manual. For a release flow,
each tagged commit should auto-build and publish `dist/qstore.html`
somewhere reproducible.

**Direction.** GitHub Actions: on tag push, run build, upload as a
release asset. Single-file artifact attached to the GitHub release.
Simple to set up, makes the deployment story clean.

### Versioning surfaced in UI
**Problem.** The package.json says `2.0.0-alpha` but it's not visible
anywhere in the running app. Users can't tell which version they're on
when reporting issues.

**Direction.** esbuild's `define` option to inject the version at build
time. Display in the Settings page footer.
